# NoteEditor PRD

> 产品需求文档 | 版本: 0.1-draft | 日期: 2026-03-22

## 1. 项目概述

### 1.1 项目名称

**NoteEditor** — NotebookLM 演示文稿 PDF 到 PPTX 的转换工具。

### 1.2 项目目标

将 Google NotebookLM 导出的演示文稿 PDF 转换为高质量的 PPTX 文件，在图片、背景、文字、布局上实现尽可能像素级的视觉还原。

### 1.3 产品定位

- **形态**：本地命令行工具，用户自部署
- **用户**：使用 NotebookLM 生成演示文稿的普通用户
- **许可**：Apache 2.0
- **运行环境**：离线优先，支持无 GPU 环境（CPU 模式），可选云端 API 作为后备

### 1.4 愿景

初期（v0.8.0 之前）专注于 NotebookLM PDF → PPTX 转换。长期目标是支持一切演示文稿样式的 PDF → PPTX 转换，成为通用的演示文稿 PDF 还原工具。

## 2. 问题分析与用户场景

### 2.1 核心问题

Google NotebookLM 能将上传的资料自动生成演示文稿（通常 12-15 页），但其 PDF 导出存在根本性缺陷：**每一页被渲染为一张完整的截图，而非结构化文档**。PDF 中没有可选文字、没有独立的图片资源、没有文字框——文本、图片、背景全部被"烘焙"进一张光栅图像中。

这导致的后果是：
- 标准 PDF 转换工具（Adobe Acrobat、Smallpdf、ILovePDF）全部失效——它们依赖 PDF 中的结构化内容
- 用户无法编辑生成的演示文稿中的任何文字
- 图片无法单独提取和复用
- 内容无法被搜索和索引

2025 年 2 月 NotebookLM 新增了原生 PPTX 导出功能，但导出的 PPTX 仍然是图片层，与 PDF 导出在本质上没有区别。

### 2.2 现有解决方案的不足

| 工具 | 类型 | 主要不足 |
|------|------|---------|
| NotebookLM2PPT | 开源 | 仅做基础背景-文字分离，文字区域使用外部 OCR API，布局还原精度有限 |
| NBLM2PPTX | 开源 | 依赖 Gemini AI 做分离，需要 Google API Key，网络依赖强 |
| Codia AI NoteSlide | 商业 | 需付费，依赖云端服务，无法离线使用 |
| Adobe Acrobat / Smallpdf | 通用 PDF 工具 | 面向结构化 PDF，对截图式 PDF 产生扭曲的布局和破碎的图形 |

**共同缺陷：** 现有工具在图片高清提取、背景精确还原方面均未达到像素级保真，且没有离线、免费、高精度同时满足的方案。

### 2.3 目标用户与场景

**目标用户：** 使用 NotebookLM 生成演示文稿后，需要获得可编辑、可复用 PPTX 的普通用户。

**典型场景：**

1. **知识分享** — 用户将研究资料上传 NotebookLM，生成演示文稿用于团队分享。导出 PDF 后希望编辑部分内容再使用，但无法编辑。
2. **学术报告** — 学生将论文资料用 NotebookLM 生成演示文稿，需要调整内容以适应不同的报告场合。
3. **内容再利用** — 用户希望提取 NotebookLM 演示文稿中的图片和文字，用于其他文档或展示。

## 3. 功能需求

### 3.1 核心功能

#### F1: PDF 输入解析

- 接收 NotebookLM 导出的 PDF 文件作为输入
- 检测并验证 PDF 格式（识别是否为截图式 PDF）
- 自动检测页面尺寸与宽高比（默认 16:9）
- 提取每页的高分辨率图像和嵌入的原始图片资源

#### F2: 布局分析

- 对每页进行语义区域检测，识别：标题、正文、图片、表格、页眉、页脚等
- 输出每个区域的边界框坐标和语义标签
- 支持中英文混合内容的布局理解

#### F3: 图片提取与还原

- 从高 DPI 渲染图中裁切图片区域，保持原始分辨率
- 优先提取 PDF 中嵌入的原始图片资源
- 按原始位置和比例放置到 PPTX 中

#### F4: 背景还原

- **视觉保真模式**：原始截图直接作为幻灯片背景，无需额外处理
- **可编辑模式**：去除文字内容，还原干净背景（程序化重建或 Inpainting）

#### F5: 文字 OCR

- 对检测到的文字区域进行 OCR 识别，提取文本内容
- 支持中英文混合、公式（LaTeX）
- 利用布局检测结果获取文字区域的位置信息
- 公式处理：OCR 输出的 LaTeX 公式渲染为图片后嵌入 PPTX（python-pptx 原生不支持 LaTeX）
- 表格处理：识别出的表格作为图片嵌入 PPTX（不还原为原生表格对象，确保视觉保真）

#### F6: 字体匹配

- 维护 NotebookLM 常用字体映射表（标题、正文、引用等）
- 根据语义标签匹配字体
- 匹配不到时回退到最接近的系统字体
- 将匹配到的字体文件嵌入 PPTX

#### F7: PPTX 生成

- 根据所选输出模式组装 PPTX：
  - **视觉保真模式**：背景图 + 提取的图片覆盖层
  - **可编辑模式**：干净背景 + 原生文字框 + 提取的图片
- 幻灯片尺寸与输入 PDF 匹配
- 输出标准 .pptx 格式，兼容 PowerPoint、Google Slides、WPS 等

### 3.2 质量优先级

按重要性从高到低排列，功能开发按此顺序推进：

| 优先级 | 质量维度 | 对应功能 |
|--------|---------|---------|
| P0 | 图片高清提取并放置在正确位置 | F3 |
| P0 | 背景和装饰元素还原 | F4 |
| P1 | 文本内容 100% 准确提取 | F5 |
| P1 | 布局/对齐精确还原 | F2, F7 |
| P2 | 字体名称/大小/颜色还原 | F6 |

### 3.3 交互功能

#### F8: CLI 命令行界面

- 基本用法：`noteeditor input.pdf -o output.pptx`
- 支持指定输出模式：`--mode visual`（默认）/ `--mode editable`
- 支持指定渲染 DPI：`--dpi 300`（默认）
- 支持指定模型运行模式：`--device gpu` / `--device cpu` / `--device api`

#### F9: 处理进度反馈

- 实时显示当前处理的页码（如 `Processing page 3/15`）
- 显示当前阶段（如 `Layout detection` / `OCR` / `Image extraction`）
- 显示预计剩余时间

#### F10: 错误处理

- 单页处理失败不阻塞整体，失败页面使用原始截图作为底板
- 处理完成后输出摘要报告：成功页数、失败页数、失败原因
- 断点续传：处理进度持久化，中断后可从断点继续

## 4. 技术方案

### 4.1 管线架构

采用 6 阶段流水线，支持两种输出模式。每阶段独立可测试，单页失败不阻塞整体处理。

```
PDF 输入
  ↓
[1] 页面渲染 ── 高 DPI 渲染 + 宽高比检测
  ↓
[2] 布局检测 ── PP-DocLayout-V3 语义区域识别（输出边界框 + 标签）
  ↓  ┌──────────────────────────┐
[3] ├→ 文字 OCR ── GLM-OCR（在布局检测裁切后的区域图上识别）
     ├→ 图片提取 ── 高 DPI 裁切 + 原始资源优先
     └→ 字体匹配 ── NotebookLM 字体映射表 + 系统字体回退
  ↓
[4] 背景提取（仅可编辑模式）── 程序化重建 / LaMA Inpainting 兜底
  ↓
[5] PPTX 重建 ── python-pptx，根据输出模式选择策略
  ↓
输出 .pptx
```

> 注：阶段 [3] 中文字 OCR、图片提取、字体匹配可并行执行。字体匹配依赖 OCR 结果确定文字区域，但可异步完成。

### 4.2 输出模式

| 模式 | 背景处理 | 文字处理 | 适用场景 |
|------|---------|---------|---------|
| **视觉保真模式**（默认） | 原始截图直接作为背景 | 保留在背景图像中（不可编辑） | 追求最佳视觉效果 |
| **可编辑模式** | 提取干净背景（去除文字） | 原生文字框覆盖（可能有轻微对齐偏差） | 需要后续编辑文字 |

视觉保真模式下无需背景提取步骤，管线更短、速度更快、视觉还原度更高。可编辑模式在布局检测 bounding box 精度范围内提供文字可编辑性，适合需要修改内容的场景。

### 4.3 技术选型

#### 页面渲染：PyMuPDF

- 从 PDF 提取每页的高分辨率图像（默认 300 DPI，可配置至 600 DPI）
- 同时提取 PDF 中嵌入的原始图片资源（如果有）
- 检测页面宽高比（NotebookLM 导出为 16:9），确保 PPTX 输出匹配
- **许可风险**：PyMuPDF 为 AGPL-3.0，与项目 Apache 2.0 不兼容。当前为个人项目、用户自部署，实际影响较低。后续如需商用，需购买 PyMuPDF 商业许可或迁移至其他方案（如 pdfplumber + pdf2image）

#### 布局检测：PP-DocLayout-V3

- Baidu 开源文档布局检测模型，Apache 2.0
- 识别语义区域：标题、正文、图片、表格、页眉、页脚等
- GLM-OCR 官方也在使用此模型作为布局检测骨干
- **输出每个区域的边界框坐标和语义标签**——这是整个管线中"位置信息"的来源，而非来自 GLM-OCR

#### 文字 OCR：GLM-OCR

- 智谱开源，0.9B 参数，Apache 2.0 许可
- OmniDocBench v1.5 得分 94.62（当前最高）
- 支持中英文混合内容、公式（LaTeX 输出）、表格
- 可本地部署（ONNX），也可调用 Zhipu BigModel API（可选云端模式）
- GLM-OCR 本身只输出文本内容，**不输出像素级位置信息或字体特征**。位置信息由上游 PP-DocLayout-V3 提供，字体匹配由独立模块处理

#### 字体匹配

NotebookLM 使用的字体种类有限，采用手动维护的映射表而非自动字体识别：

1. **NotebookLM 字体映射表** — 手动收集 NotebookLM 各模板中标题、正文、引用等文本类型对应的字体，维护为配置
2. **匹配逻辑** — 根据布局检测的语义标签（标题/正文/引用）查映射表，精确匹配字体
3. **回退策略** — 映射表无匹配时，按视觉风格选择最接近的系统字体（如 Google Sans → Arial）
4. **字体嵌入** — 匹配到的字体文件嵌入 PPTX，确保跨设备渲染一致

#### 图片提取

- 从高 DPI 页面渲染图中，按布局检测的边界框裁切图片区域
- 如果 PDF 中存在嵌入的原始图片资源（PyMuPDF 可提取），优先使用原始资源以获得更高分辨率
- 保持图片的原始比例和位置信息

#### 背景提取（仅可编辑模式）

可编辑模式需要从原始截图中去除文字，还原干净背景。采用从简单到复杂的分层策略：

| 策略 | 适用场景 | 方法 |
|------|---------|------|
| 程序化背景重建 | 纯色或渐变背景 | 用布局检测 mask 覆盖文字区域，对非文字区域取颜色中值/渐变拟合还原 |
| LaMA Inpainting | 复杂图案或照片背景 | 用布局检测的 mask 标记文字区域，AI 填充还原（Apache 2.0） |

视觉保真模式下完全跳过此阶段。

#### PPTX 重建：python-pptx

MIT 许可，Python 生态中 PPTX 创建的标准库。幻灯片尺寸设为 16:9（标准宽屏），与 NotebookLM 导出一致。

**视觉保真模式：**
1. 原始截图设为幻灯片背景图像（像素级保真）
2. 提取的图片按原始位置覆盖（分辨率可能优于原图中的嵌入版本）
3. 不放置文字框（文字已包含在背景图像中）

**可编辑模式：**
1. 干净背景设为幻灯片背景图像
2. 提取的图片按原始位置放置
3. OCR 识别的文本以原生文字框放置，使用匹配的字体、大小、颜色（bounding box 来自 PP-DocLayout-V3）

### 4.4 运行时适配

| 硬件条件 | 策略 |
|---------|------|
| GPU（8GB+ VRAM） | 全部模型 GPU 推理，速度优先 |
| 无 GPU（纯 CPU） | ONNX Runtime CPU 模式，速度较慢但功能完整 |
| 资源极度受限 | 可选调用 GLM-OCR 云端 API 作为后备 |

### 4.5 模型管理

- 所有模型默认离线运行，README 提供模型下载和存放指引
- 首次运行时自动检测模型文件是否存在，缺失时提示用户下载
- 模型存放于项目固定目录，支持统一管理

### 4.6 处理健壮性

- **单页隔离** — 任意一页处理失败不影响其余页面，失败页面使用原始截图作为底板
- **断点续传** — 页级别持久化，记录已完成的页码，中断后跳过已处理页继续
- **进度显示** — CLI 实时输出当前处理页码、阶段、预计剩余时间

## 6. 非功能性需求

### 6.1 性能

| 指标 | 目标 |
|------|------|
| 15 页 PDF 处理时间（GPU） | ≤ 3 分钟 |
| 15 页 PDF 处理时间（CPU） | ≤ 30 分钟 |
| 单页渲染分辨率 | 300 DPI（默认），支持 600 DPI |
| 内存占用峰值（GPU） | ≤ 8 GB VRAM |
| 内存占用峰值（CPU） | ≤ 16 GB RAM |

### 6.2 易用性

- **安装**：`pip install` 一键安装，README 提供模型下载和存放的详细指引
- **使用**：单条命令完成转换，无 GUI，无复杂配置
- **反馈**：处理过程实时显示进度，出错时给出明确的错误信息和解决建议

### 6.3 可维护性

- 管线各阶段模块化，独立可测试
- 模型与业务逻辑解耦，模型可独立替换升级
- 技术栈跟随依赖项目的最佳实践，不人为绑定特定框架

### 6.4 可移植性

- 纯 CPU 环境可运行（速度较慢）
- 支持 Windows、macOS、Linux
- 不依赖特定 GPU 品牌（CUDA / Metal / CPU 自适应）
