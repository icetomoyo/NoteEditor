## 017: OCR 推理后端重构（Transformers / Ollama / vLLM）

**Category:** Refactor | **Priority:** Critical | **Status:** Completed | **Started:** 2026-04-11

### 描述

重构 OCR 推理架构，将不可用的 ONNX single-call 路径替换为正确的 VLM 推理方式。实现 `OCRBackend` 抽象层，支持 Transformers（进程内）、Ollama（本地服务）、vLLM（高性能服务）三种本地推理后端，保留已有的 Zhipu API 云端后端。

### 设计要点

- 参考 DD §2.3（已更新）、DD §3.5
- GLM-OCR 是 VLM（CogViT 0.4B + GLM-0.5B），需要自回归 token 生成
- `OCRBackend` Protocol 定义统一的 `recognize(image, task) → OCRResponse` 接口
- 四种实现：TransformersBackend / OllamaBackend / VLLMBackend / ZhipuAPIBackend
- `--device` CLI 参数路由到对应后端
- `auto` 模式按优先级探测可用后端

### 影响范围

**需要创建的文件**:
- `src/noteeditor/infra/ocr_backend.py` — Protocol + 四种 Backend 实现 + 工厂函数

**需要修改的文件**:
- `src/noteeditor/stages/ocr.py` — `extract_text()` 接受 `OCRBackend` 替代 `ort.InferenceSession`，删除 ONNX 推理代码
- `src/noteeditor/infra/model_manager.py` — 添加 `create_ocr_backend()` 方法，删除 `get_ocr_model()`
- `src/noteeditor/infra/config.py` — `PipelineConfig` 添加 `device` 字段
- `src/noteeditor/pipeline.py` — 使用 `create_ocr_backend()` 替代 `get_ocr_model()`
- `src/noteeditor/cli.py` — `--device` 选项更新
- `pyproject.toml` — 添加 `transformers` 可选依赖

**需要更新的测试**:
- `tests/unit/test_ocr.py` — mock OCRBackend 替代 mock ONNX session
- `tests/unit/test_model_manager.py` — 更新
- `tests/unit/test_pipeline_v2.py` — 更新

### 验收标准

- [ ] `OCRBackend` Protocol 定义清晰，四种实现可独立测试
- [ ] `--device transformers` 使用 HuggingFace Transformers 推理
- [ ] `--device ollama` 调用本地 Ollama 服务（需服务已启动）
- [ ] `--device vllm` 调用本地 vLLM 服务（OpenAI 兼容 API）
- [ ] `--device api` 调用 Zhipu 云端 API（保持已有行为）
- [ ] `--device auto` 按优先级探测可用后端
- [ ] 后端不可用时抛出明确错误信息和解决建议
- [ ] ONNX OCR 代码完全移除（布局检测保留 ONNX）
- [ ] 所有 OCRResult 保持 frozen dataclass

## 018: Builder 文本框填充策略修复

**Category:** Fix | **Priority:** Critical | **Status:** Planned

### 描述

修复 Builder 中文本框始终使用白色填充的问题。当背景提取阶段已生成干净背景时，文本框应使用透明填充；仅在回退到截图背景时使用白色填充。

### 影响范围

**需要修改的文件**:
- `src/noteeditor/stages/builder.py` — `_add_text_box()` 和 `build_editable_pptx()`

**需要更新的测试**:
- `tests/unit/test_builder_editable.py`

### 验收标准

- [ ] 有 background_image 时文本框透明，文字直接显示在干净背景上
- [ ] 无 background_image 时文本框白色填充（与 v0.4.0 行为一致）
- [ ] 背景图使用 PPTX 原生背景 API，不可被用户误移动
- [ ] fallback 页面仍使用截图 + 白色文本框

## 019: 多行文本字号估算修复

**Category:** Fix | **Priority:** Critical | **Status:** Planned

### 描述

修复字号估算对多行 BODY_TEXT 区域严重高估的问题。结合 OCR 文本内容的行数修正 bbox 高度公式。

### 设计要点

- 参考 DD §2.8（已更新）
- 修正公式：`font_size_pt = (bbox_height / line_count) * (72 / dpi) * 0.8`
- `line_count = max(1, text.count('\n') + 1)`
- `estimate_styles()` 需要接收 OCR 结果作为额外输入

### 影响范围

**需要修改的文件**:
- `src/noteeditor/stages/style.py` — `estimate_styles()` 签名变更，接收 OCR 结果
- `src/noteeditor/pipeline.py` — 传递 OCR 结果给 style 阶段

**需要更新的测试**:
- `tests/unit/test_style.py`

### 验收标准

- [ ] 单行 TITLE 区域字号估算与 v0.4.0 一致
- [ ] 多行 BODY_TEXT（如 5 行）字号 ≈ 单行估算值 / 5
- [ ] OCR 文本无换行时退化为 line_count=1

## 025: LaMA Inpainting（复杂背景）

**Category:** New | **Priority:** Medium | **Status:** Planned

### 描述

集成 LaMA Inpainting 模型，作为复杂背景（std >= 50）的填充策略，替代当前的白色填充回退。LaMA 作为可选依赖，不可用时回退到白色填充。

### 设计要点

- LaMA（Apache 2.0），ONNX 格式
- 输入：原始图像 + 文字区域 mask → 输出：AI 填充后的图像
- 作为 `_fill_complex()` 替代 `_fill_fallback()`
- 模型存放于 `~/.noteeditor/models/lama.onnx`

### 影响范围

**需要修改的文件**:
- `src/noteeditor/stages/background.py` — 添加 `_fill_complex()` LaMA 逻辑
- `src/noteeditor/infra/model_manager.py` — 添加 LaMA 模型加载

### 验收标准

- [ ] 复杂背景使用 LaMA 填充，视觉质量优于白色填充
- [ ] LaMA 模型不存在时回退到白色填充（不报错）
- [ ] LaMA 推理失败时回退到白色填充

### 验收标准
